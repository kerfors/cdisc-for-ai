Part 1
The mismatch — what an SoA burden calculator needs vs what XRAYCHEST resolves to

A Schedule of Activities binds each study activity to a standardised concept (ideally a COSMoS DSS). A burden calculator then reads burden-relevant attributes off that concept — duration, discomfort, preparation, patient effort, logistics — and sums across the timeline.

For chest X-ray, XRAYCHEST is the DSS. But the burden profile of a chest X-ray varies materially with acquisition conditions the DSS does not model. The clearest case: patient position.

Standing PA chest X-ray vs supine portable chest X-ray — same XRAYCHEST DSS, different real-world activity:

  • Standing PA: ambulatory patient walks to imaging suite, ~5–10 min door-to-door, single PA + lateral view, minimal staff assist, patient actively positions.
  • Supine portable: bedside acquisition, longer setup, single AP view, higher staff effort, lower patient effort but typically a sicker patient with additional disruption to rest.

A burden calculator that treats both as "XRAYCHEST" hides the distinction. For a study recruiting hospitalised patients where supine imaging is protocolised, this is not a marginal error.

Part 2
The DSS slot inventory — what is modelled, what is not

The flattened DSS schema (as carried in the cdisc-for-ai interim file) models five qualifier-like slots at the DSS level: Specimen, Method, location, laterality, evaluator, plus unit and decimal-place metadata. There is no position column in the flattened schema.

At VLM-row level (the un-flattened DSS source) --POS DOES appear — in 40 rows total: 31 EGPOS rows across EG DSSs (all with codelist C71148, no value_list restriction), 9 VSPOS rows across VS DSSs (with codelist C71148 and typically a restricted value_list like PRONE;SEMI-RECUMBENT;SITTING;STANDING;SUPINE for DIABP/HR/PULSE). Zero rows in any PR, MK, TR, TU DSS.

So the gap is not DSS-wide — it is imaging-specific. Position has been curated for blood-pressure and ECG measurements, where it was identified as clinically discriminating. It has not been curated for imaging procedures or imaging-derived measurements, where it is equally discriminating for burden and data interpretation.

Identity layer rich, measurement-spec layer uneven

This is a layered asymmetry with domain-specific coverage. The identity layer (SDTM CT + NCIt) publishes a structured POSITION vocabulary with 17 governed terms, each with its own NCIt concept code. The measurement-spec layer (COSMoS DSS) binds POSITION in EG and VS — but not in any PR, MK, TR, or TU DSS, where clinical relevance for imaging is equally strong.

A downstream tool wanting to use "standing chest X-ray" as an anchor can name that concept (governed terms exist for both pieces), but cannot resolve it to a single DSS identifier the way it can for, say, "sitting blood pressure" (covered by a VS DSS with VSPOS as a slot).

Part 3
Case specialisations — what SoA design authors on top of XRAYCHEST

The same pattern the Glucose Study-Intent Story describes for lab tests applies here: a study designer does not author XRAYCHEST; they author cases that specialise it. For burden-aware SoA design, the specialisations pin position and optionally view.

The case_id values (ID101, ID102, ID103) are syntactic placeholders — the architectural point is that such IDs would need to be governed by a layer that does not yet exist for imaging cases.

Part 4
Four architectural options for closing the gap

If a burden-aware SoA model needs to discriminate standing vs supine chest X-ray, four structural options exist. Each has clear trade-offs.

Option 1
Proliferate DSSs
Publish XRAYCHEST_STANDING, XRAYCHEST_SUPINE, XRAYCHEST_SEMIRECUMBENT, … as separate COSMoS DSSs. Each has its own vlm_group_id.
Trade: Violates the COSMoS principle that DSSs model recording templates, not medical ontology. Scales poorly — every position-sensitive procedure multiplies. Cross-study interoperability at coarser grain (all chest X-rays) gets harder.
Option 2
Composite key at SoA / USDM level
Activity = tuple (dss_ref, position, view, …). USDM's Activity extended with burden-relevant qualifier attributes referencing governed SDTM CT codelists.
Trade: Clean architecturally — identity layer stays published, DSS stays single, burden lives where activities are. Requires a USDM extension (or sponsor layer over USDM). Cross-sponsor vocabulary only if the extension is standardised.
Option 3
Sponsor-defined qualifier on the Activity
Each sponsor adds a position attribute to their internal Activity model, referencing CDISC CT POSITION values but authored in sponsor-local catalogues.
Trade: Pragmatic today — usable without standards changes. Opaque to tools across sponsors. Burden profiles cannot be compared at the case level without translating sponsor catalogues.
Option 4
Add position as a DSS schema slot
COSMoS adds position alongside location, laterality, evaluator as a first-class DSS qualifier slot, bound to POSITION codelist (C71148). Imaging DSSs populate it as a value_list or assigned term where relevant.
Trade: Correct long-term fix, structurally symmetric with laterality. Multi-quarter effort at CDISC COSMoS / DDF. Does not solve view/contrast/preparation — each additional qualifier is its own extension.

Near-term pragmatic path (while Option 4 is pursued as the long game): Option 3 with an eye on convergence to Option 2. A sponsor defines burden-aware case specialisations that pin (dss_ref, position, view), using CDISC CT terms for the qualifier values. When USDM or COSMoS grows a standard slot, the sponsor mappings migrate cleanly because the values are already governed identifiers — only the attribute-to-slot binding changes.

Summary
Why this case matters for the cdisc-for-ai three-layer model

This is a concrete example of a recurring architectural asymmetry: the identity layer outreaches the measurement-spec layer.

SDTM CT publishes a governed POSITION codelist with 17 terms. --POS is populated in some DSSs (EG, VS — 40 rows total). The vocabulary, the variable, and even DSS-level bindings exist. But the COSMoS DSSs that downstream tools (SoA designers, burden calculators, CRF libraries) would want to bind to for imaging — XRAYCHEST, CTSCANCHEST, MK Sharp/Genant — have no such binding.

The same pattern applies to other qualifier dimensions that vary clinically but are not in the DSS schema: acquisition view (PA/AP/lateral), fasting state for imaging, timing relative to challenge events, inpatient-vs-outpatient setting. Each is a case where a well-governed identity fact is unreachable from the measurement-spec layer.

For the project's three-layer analytical model, this argues for explicit qualifier-dimension tracking — not just test-code identity — as a first-class artefact. The traversable graph should be able to answer: "for this DSS, which SDTM-CT-governed qualifier codelists are attachable, and which are pinned, and which are silent?" Today that question is answered by reading the DSS XLSX row by row, as this analysis had to do.

Registry need
A burden-aware case-specialisation registry over DSS

Chest X-Ray Standing, Chest X-Ray Supine, Chest X-Ray Semi-Recumbent — three distinct SoA activities with three distinct burden profiles, but one DSS (XRAYCHEST) and one BC. The pieces a registry would need to compose are already governed:

— Procedure identity: XRAYCHEST (COSMoS DSS vlm_group_id), X-RAY from PROCEDUR (C101858), anchor NCIt C38101.
— Position value: governed CDISC CT POSITION (C71148) terms — STANDING (C62166), SUPINE, SEMI-RECUMBENT (C111310), each with an NCIt code.
— SoA activity reference: USDM Activity object (identifier exists per protocol, not cross-sponsor).

What is missing is the governed composition — a registry entry that says "case ID101 = XRAYCHEST + STANDING, burden-profile standard-outpatient, evidence-link <ref>". The syntactic IDs above mark where such governed identifiers would go.

This is the same architectural slot the Glucose Study-Intent Story identifies for OGTT/FPG cases. The imaging case shows the pattern generalises beyond specimen-based Findings.
